# 이벤트 기반 아키텍처 도입 분석

## 결론: 지금 이벤트 기반 아키텍처는 필요 없습니다

---

## 현재 프로젝트 현황

| 항목 | 상태 |
|------|------|
| 아키텍처 | 단일 모놀리스 (Spring Boot) |
| 코드 규모 | ~873줄 Kotlin, MVP 단계 |
| 도메인 | 충전 / 지갑 / 기프티콘 (강하게 결합) |
| 동시성 처리 | DB 레벨 조건부 UPDATE + 낙관적 락 |
| 멱등성 | Unique Constraint + Idempotency-Key |

---

## 이벤트 스토밍에서 보이는 것

이벤트 스토밍 결과 이미 이벤트 흐름이 잘 정의되어 있다:

```
충전요청됨 → (PG 성공여부) → 충전 PG승인됨 → 지갑잔액 증가됨 → 지갑원장 생성됨
구매요청됨 → (잔액 차감 가능여부) → 구매확정됨 → 기프티콘 생성됨 → 지갑 차감됨 → 지갑원장 생성됨
```

이벤트 스토밍 결과는 훌륭하지만, **이 이벤트들이 전부 하나의 트랜잭션 안에서 일어나야 하는 것들**이다. 이게 핵심이다.

---

## 왜 EDA가 지금 맞지 않는가

### 1. 트랜잭션 일관성이 핵심 요구사항

기프티콘 구매 하나가 `Wallet 차감 + Gifticon 발급 + Ledger 기록`을 단일 ACID 트랜잭션으로 묶어야 한다. 이걸 이벤트로 분리하면:
- Wallet은 차감됐는데 Gifticon이 안 생겼으면? → 보상 트랜잭션(Saga) 필요
- 이게 현재 DB 레벨 조건부 UPDATE보다 훨씬 복잡하고 버그가 많이 난다

### 2. 현재 동시성 해결책이 이미 훌륭함

```sql
UPDATE wallet SET balance = balance - :amount 
WHERE user_id = :userId AND balance >= :amount
```

이 단순한 쿼리 하나가 EDA + 분산 락을 대체하고 있다. 굳이 메시지 브로커를 넣을 이유가 없다.

### 3. 도메인들이 강하게 결합되어 있음

이벤트 스토밍에서 `충전 PG승인됨 → 지갑잔액 증가됨 → 지갑원장 생성됨` 이 세 개는 독립적으로 실패해선 안 되는 연산들이다. EDA의 "eventually consistent"는 여기서 독이 된다.

### 4. 현재 복잡도가 정당화되지 않음

Kafka/RabbitMQ 도입 = 브로커 운영 + Outbox 패턴 + 멱등 소비자 + Dead Letter Queue + 메시지 순서 보장... MVP에서 이 비용을 지불할 이유가 없다.

---

## 그렇다면 언제 EDA로 가야 하나

아래 시나리오가 생기면 그때 고려한다.

**① 팬아웃 필요 시**
충전 완료 후 푸시알림, 포인트 만료 경고, 통계 집계 등 여러 시스템이 하나의 이벤트를 소비해야 할 때. 지금처럼 단일 수신자라면 EDA가 과잉이다.

**② 외부 시스템 연동이 늘어날 때**
실제 PG사, CRM, 정산 시스템 등 외부 시스템들이 결합되기 시작하면 이벤트 버스로 분리하는 게 유리하다.

**③ 처리량이 DB 동기 처리 한계에 닿을 때**
초당 수천 건의 충전 요청이 오는 상황에서 DB 병목이 생기면 비동기 큐 처리를 고민하면 된다.

**④ 감사/이벤트 소싱이 법적 요건이 될 때**
금융 규제에서 "모든 상태 변화를 이벤트로 남겨라"는 요구가 생길 때.

## MSA는 언제?

MSA 전환의 트리거는 기술이 아니라 **팀과 조직**이다.

| 상황 | 판단 |
|------|------|
| 팀이 2-3명 | 모놀리스 유지 |
| 팀이 10명+, 도메인별로 나뉨 | 모듈형 모놀리스 고려 |
| 여러 팀이 같은 레포에서 배포 충돌 | MSA 전환 시점 |
| 도메인별 스케일 요구사항이 달라짐 | MSA 전환 시점 |

이 프로젝트의 경우:
- **지금**: 모놀리스 유지, 패키지 구조만 도메인별로 명확히 분리
- **1단계**: `충전 / 지갑 / 기프티콘`을 **모듈형 모놀리스**로 경계 강화 (서로 직접 DB 접근 금지, 서비스 인터페이스로만 소통)
- **2단계**: 실제 배포/스케일 병목이 생기는 도메인만 먼저 분리

---

## 현재 가장 현실적인 설계 결정

이벤트 스토밍의 trade-off 메모 — **"구매요청 결과로 기프티콘이 나오면 어디에 기록할 것인가?"** — 이게 제일 현실적인 설계 결정이다. EDA 없이도 해결 가능하고, 해결해야 하는 문제다.

현재 코드는 구매요청 테이블 없이 Gifticon에 바로 저장하는 방식인데, 구매 이력 추적이 필요하다면 `PurchaseRequest` 애그리거트를 별도 테이블로 분리하는 게 맞다.
